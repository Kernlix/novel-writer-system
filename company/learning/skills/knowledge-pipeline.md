# 🔗 知识管线Skill (Knowledge Pipeline)

> 打通学习分析到写作能力的最后一公里——从 analysis 文件中自动提取技法、生成可部署的 Skill、完成注册，让学习成果「活在写作流程中」。

## 命令

```
/novel:learn:pipeline
```

## 功能

自动化执行「分析→技法提炼→Skill生成→Agent注册」的全流程。

## 前置条件

- `knowledge/learned/` 下存在分析结果文件（至少1个目录）
- `company/REGISTRY.md` 可读可写

## 执行流程

### 第1步：扫描分析目录

```bash
# 扫描所有分析结果目录
find knowledge/learned/ -maxdepth 1 -type d | sort
```

遍历每个子目录中的 `.md` 文件，识别分析报告特征：
- 包含「维度」「分析」「技法」等关键词
- 文件大小 > 1KB（排除空文件/索引文件）
- 文件名包含「卷」「章」「分析」等标记

### 第2步：逐文件提取核心技法

对每个分析文件，执行三轮提取：

```
第1轮 — 全文扫描
  └── 识别所有提到的技法、手法、技巧

第2轮 — 维度过滤
  └── 筛选在≥2个维度中被提及的技法

第3轮 — 可操作性评估
  └── 将技法转化为「写手可直接应用」的步骤
```

提取格式：
```
[技法名称] | 来源: 《作品名》第N卷 | 维度: [维度1, 维度2]
可操作性: ★★★★☆
通用性: project / global
步骤:
  1. ...
  2. ...
  3. ...
```

### 第3步：去重 & 聚类

```
已有Skill列表 ← company/writing/skills/ + company/learning/skills/
                                      ↓
              新提取技法 ──→ 与已有Skill比对
                                      ↓
                        ┌── 高度重叠(≥80%) → 跳过
                        ├── 部分重叠(40-80%) → 合并到已有Skill
                        └── 无重叠(≤40%) → 新建Skill
```

聚类规则：
- 同 domain 技法归入同一Skill（如"悬疑技巧"、"角色弧光"）
- 每个Skill包含3-8条相关技法
- 超过8条则拆分为两个Skill

### 第4步：生成Skill文件

按以下模板生成：

```markdown
---
id: <auto-generated-slug>
name: <从技法提炼的名称>
category: <按目标部门分类>
command: /novel:write:<domain>
description: <一句话描述，≤50字>
source: <来源作品列表>
created: <当前日期>
---

# <Skill标题>

> 从《<作品名>》分析中提炼的<技法类型>技法

## 命令
`/novel:write:<domain>`

## 适用场景
- 章节类型：<如：悬疑揭秘、情感高潮、战斗场景>
- 情感基调：<如：紧张、温馨、热血>
- 视角要求：<如：限制视角、全知视角>

## 核心技法

### 技法1: <名称>
**来源**：《<作品>》第N卷
**步骤**：
1. ...
2. ...
3. ...

**示例**：
> <从分析结果中提取的具体段落>

### 技法2: <名称>
...

## 常见陷阱
- ...

## 检查清单
- [ ] 本章是否应用了至少1条本Skill中的技法？
- [ ] 技法是否自然融入叙事，而非生硬插入？
```

写入路径判断：
| 技法 domain | 目标目录 |
|:------------|:---------|
| scene / dialogue / writing / style | `company/writing/skills/` |
| comedy / humor | `company/writing/skills/` |
| character / emotional / arc | `company/writing/skills/` |
| plot / rhythm / suspense / mystery | `company/writing/skills/` |
| review / check / quality | `company/review/skills/` |
| 通用 / 跨域 | `company/learning/skills/` |

### 第5步：注册到系统

#### 5a. 更新 REGISTRY.md

在 `company/REGISTRY.md` 对应部门的 `**Skills:**` 行中追加新Skill ID：

```
原：**Skills:** `chapter-writing`, `worldbuilding`, ...
新：**Skills:** `chapter-writing`, `worldbuilding`, ..., `new-skill-id`
```

#### 5b. 更新 writer-agent.md（写作类Skill）

在 `company/writing/writer-agent.md` 的「核心参考技法」表中新增行：

```
| <domain> | <Skill名称> | <command> | <适用场景> |
```

#### 5c. 更新 skill-matcher-agent.md

在 `company/writing/skill-matcher-agent.md` 的章节类型映射表中新增映射。

### 第6步：输出部署报告

```
📦 知识管线执行完成

扫描目录：knowledge/learned/无职转生/, knowledge/learned/夏日重现/
分析文件：26 + 13 = 39 个文件

提取技法：XX 条
  ├── P0 (多作品共有): X 条
  ├── P1 (单作品核心): X 条
  └── P2 (高维度密度): X 条

去重结果：
  ├── 跳过 (已存在): X 条
  ├── 合并 (部分重叠): X 条
  └── 新建: X 条

新增Skill：
  ✅ <skill-id-1> — <描述> → company/writing/skills/
  ✅ <skill-id-2> — <描述> → company/writing/skills/
  ...

已更新文件：
  ✅ company/REGISTRY.md — Skills列表更新
  ✅ company/writing/writer-agent.md — 核心参考技法表更新
  ✅ company/writing/skill-matcher-agent.md — 章节类型映射更新
```

## 参数选项

| 参数 | 说明 | 默认值 |
|:-----|:-----|:-------|
| `--dir <path>` | 指定分析结果目录 | `knowledge/learned/` |
| `--force` | 强制重新生成（忽略已存在Skill） | `false` |
| `--dry-run` | 仅预览，不实际写入文件 | `false` |
| `--min-dimensions <n>` | 技法最少出现的维度数 | `2` |
| `--max-skill-size <n>` | 单个Skill最多包含的技法数 | `8` |

## ⚠️ 注意事项

1. **不要修改已有Skill的核心逻辑**——只做增量添加，合并时保留原有内容
2. **Skill命名规范**：小写英文+连线符，如 `suspense-layering`、`character-arc-pacing`
3. **避免「花瓶Skill」**——每个Skill必须有具体的、可操作的步骤，不能只是「感悟」
4. **source字段必填**——标注技法来源作品，便于追溯和验证
5. **合并时保留最高置信度版本**——如果同名技法在多个分析中出现，取描述最详细的版本

## 与其他Skill的关系

| Skill | 关系 | 说明 |
|:------|:-----|:-----|
| `multi-agent-learning` | 上游 | 多Agent并行分析产出分析结果 |
| `story-deconstruction` | 上游 | 拆文四步法产出分析结果 |
| `external-study` | 上游 | 外部学习产出分析结果 |
| `internal-analysis` | 上游 | 内部分析产出分析结果 |
| `style-learning` | 可合并 | 风格学习Skill可合并到本管线的输出 |
| `skill-self-improvement` | 互补 | 本管线处理外部学习，self-improvement处理内部优化 |

## 验证步骤

执行完成后，验证以下内容：
1. 所有新Skill文件物理存在且格式正确
2. REGISTRY.md 中新增的Skill ID无拼写错误
3. writer-agent.md 的引用路径正确
4. 无文件内交叉引用断裂
5. `pre-commit-check.sh` 通过（如有）
